## Decisions

### D1 — Intent and working mode
- **Intent:** Create (no prior brief existed).
- **Mode:** Coaching path (walk section by section, push on thin assumptions).

### D2 — Product boundary
- This remains Zac's **personal** CV / contract-and-recruiter-winning site.
- A separate **We Right Code B2B** site (for true B2B contracts and product marketing)
  is a **future, separate build** — explicitly out of scope here.

### D3 — Project 1 reframed: migrate, not upgrade-in-place
- Original plan was "upgrade all out-of-date packages on Gatsby in place."
- Reframed to **migrate off Gatsby** after evidence that Gatsby is in maintenance
  limbo since the Netlify acquisition (Feb 2023): engineering headcount cut, active
  development largely halted, ecosystem consensus is "don't start new projects on it."
  Sources: GitHub "Is GatsbyJS Officially Dead?" discussion #39062; Smashing Magazine
  "The End of My Gatsby Journey" (2024); Bejamas "Gatsby is going down."
- **RESOLVED → Next.js + TypeScript.** Facilitator initially leaned Astro (right-
  tool-for-the-job + native markdown + TS-first). Zac chose **Next.js** on stronger
  business grounds: Next.js job-market demand vastly exceeds Astro's, and a core *job*
  of this site is to demonstrate in-demand skills and give Zac that experience — so
  marketability of the skill outweighs tool-purity for this product. Markdown→HTML is
  a solved problem in the Next ecosystem (MDX / next-mdx-remote / react-markdown), so
  the backroom is no harder for it. Facilitator conceded on the merits.
- Rationale Zac endorsed: migrating a dying framework to a modern stack *is* the
  System Modernisation service line — the work becomes the case study.

### D4 — Modernisation produces visible documentation artifacts
- The migration will deliberately produce ADRs, research documents, and reasoning that
  explain changes, show where pragmatism was applied and why, and demonstrate that
  delivery was kept as the top priority throughout. (BMAD generates much of this as a
  byproduct of doing the work properly.)

### D5 — New feature: the "backroom"
- A deliberately opt-in section of the site that renders the modernisation markdown
  docs to HTML, with prominent links to the GitHub repo.
- **Design principle:** opt-in technical depth, NOT a merger of audiences. Front-of-
  house stays recruiter-flashy; the backroom is a side-door technical evaluators
  choose to enter. Avoids diluting the main recruiter flow.
- Reframes the current split (repo = technical audience, site = non-technical) toward
  the site being a fuller one-stop shop — without dissolving audience segmentation.

### D6 — Two projects, and what falls in each (RESOLVED)
- **Project 1 — Modernisation:** Gatsby → **Next.js + TypeScript** migration. Produces
  the modernisation documentation as a byproduct (see D7).
- **Project 2 — Content & showcase:** content refresh (missing jobs, updated CV +
  photo, parity with current CV) **and** the backroom build + console easter egg (D8).
- Zac runs these as **separate projects** to keep focus on each. The backroom sits in
  Project 2 (it is content-driven), NOT as a capstone of Project 1 — Q2 resolved.

### D7 — Documentation: two homes, one-way flow
- **`_bmad-output/`** — raw process/audit artifacts, generated as work happens.
- **`docs/`** — a **curated, public-facing** document structure *derived* from the
  process docs, polished for wider circulation and free to deviate in depth/emphasis
  to put Zac's best foot forward. This is what the md→HTML package renders into the
  backroom.
- **Principle:** `docs/` is **derived and curated, not a mirror** — process docs are
  the source, public docs are the edit. One-way street (avoids a sync chore).
- Fits the repo grain: `project-context.md` already designates `docs/` as the project
  knowledge folder.

### D9 — Defer public-facing doc generation to Project 2 (pragmatism)
- Project 1 captures **raw** decision/process data only — to a standard useful for the
  active work and forming a solid base. It does **not** produce the polished, curated
  public docs.
- Translating raw → curated portfolio documentation happens in **Project 2**, against a
  settled foundation, to avoid editing everything twice and to streamline Project 1
  delivery.
- **Meta-note:** this deferral is itself an exemplar pragmatism decision ("protect
  delivery velocity by deferring non-blocking polish") — capture it as documented
  content destined for the backroom.

### D10 — Audience priority and the core-function insight
- **Primary customer: the agency recruiter** — opens the site, says "cool," calls.
  For them the site is a fast credibility gate + CV grab.
- **Technical evaluators** linger longer but are also won mainly by the **wow factor /
  front-of-house polish**, not by poring over detail.
- **Insight:** both audiences are served by essentially the **same core function** —
  the slick, high-impact front-of-house impression. The backroom deepens the technical
  evaluator's "wow" but is a **differentiator/bonus, not load-bearing** for winning the
  contract.
- **Consequence to reflect in the brief (Problem / Success Criteria):** the migration
  (Project 1) is largely **invisible to the primary customer**, who responds to visual
  polish the current site already delivers. Project 1's real payoff is therefore
  credibility-hygiene, marketable skills (Next/TS), the modernisation case study, and a
  clean foundation for Project 2 — NOT "more recruiter callbacks." To be stated
  honestly rather than fudged. (See OPEN QUESTION Q3.)

### D8 — Backroom UX (Project 2)
- A **"More interested in how this site is built?"** link — present but unshowy
  (deliberate side-door; reaffirms D5's opt-in-depth principle).
- A **console ASCII-art easter egg** shown when technical users open dev tools, with
  clickable links to the backroom and the site's GitHub repo (tech-recruiting Easter-
  egg tradition — GitHub/Stripe style).

### Q3 — Sequencing flag (RESOLVED → trajectory holds)
- Flag: primary customer (recruiter) responds to visual flash the current site already
  delivers; the downloadable CV is also somewhat stale → possible case for fixing
  content before the migration.
- Resolution: the CV gap is **not actively harmful** and is **covered by other channels**
  (LinkedIn + sending updated CVs directly); Zac has maintained the PDF incrementally
  over the years (visible in git log). No revenue emergency → **Project 1 (migration)
  first stands.** Decision made with eyes open.
- Wrinkle banked for Project 2 Problem framing: the site has quietly degraded from
  "canonical always-current CV home" to a fallback (Zac routes around it). Project 2
  partly **restores the site as the canonical, current home**, not merely "freshens"
  content.

### D11 — Success is efficient delivery, not metrics
- **No hard success tracking** (page visits, engagement, conversion). Signals would be
  weak — no marketing/acquisition push; the site is "a stop along the way to a
  contract," not a destination.
- **Google Analytics (gtag):** keep it working (low effort); Zac will glance in the
  first few months out of interest only. **Not a KPI.**
- **The real success metric:** both projects **delivered to a high standard in a
  reasonable time**, so Zac can get back to the actual revenue activity — hunting
  contracts. Time-boxed, efficient delivery is the headline criterion.
- **Implied guardrail:** over-investment / gold-plating (incl. the backroom + easter-
  egg rabbit holes) is itself a failure mode. Brief must guard against scope creep.
