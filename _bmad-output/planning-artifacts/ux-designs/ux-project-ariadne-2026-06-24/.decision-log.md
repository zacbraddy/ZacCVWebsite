### 2026-06-24 — Backroom core IA: wiki with left nav (Zac brain dump)

Zac's opening vision for the Backroom:

- **Layout model:** a wiki. Left **nav bar** listing each ADR by its **number + heading**; main pane renders the selected doc.
- **Nav items are not plain links** — each is a styled "element" (drawer of elements, "not cards but elements") that presents the ADR number + heading in a pleasing way. Reference: the LinkedIn job-list rows (saved to `imports/linkedin-job-list-reference.png`) — a tile/badge on the left, strong title, quiet supporting metadata, subtle row separators, one row gently highlighted as selected. **In our palette, not LinkedIn's.**
- **Overview page = landing page** of the Backroom, sits at the top of the nav. Contains prose explaining what the Backroom is, what's here, and what Zac hopes to demonstrate to the reader — explicitly framed as **a sales pitch** highlighting the positive things the ADRs show about him (judgement, trade-off reasoning, pragmatism).

**Palette (from `src/app/globals.css`):**

- Dark (default): bg `#333`/`#555`, accent-secondary cyan `#04b4e0`, tertiary gold `#e0b404`, text `#fafafa`.
- Light: bg `#eee`/`#ddd`, accent-secondary blue `#3058b5`, tertiary terracotta `#e6593d`, text `#333`.
- Fonts: Roboto (`--font-sans`), Permanent Marker (`--font-fancy-heading`). xs breakpoint 410px.

**Why:** content-first wiki reading model fits "calmer than front-of-house, content is the star." The LinkedIn-row treatment makes the nav itself feel considered without competing with front-of-house flash.

### 2026-06-24 — Backroom shell, nav-row content, doc grouping (3 decisions)

1. **Dedicated Backroom shell.** Entering the Backroom REPLACES the front-of-house left sidebar (portrait + Home/About/Resume/Content). The Backroom is its own two-pane space: **ADR/doc nav (left) | doc content (right)**, with a low-key **"◀ back to the site"** affordance at the top of the nav. This is the "side door into a different, calmer room" — a deliberate threshold, not just another page. Front-of-house flash untouched; Backroom reads as a distinct competent-and-candid space.

2. **Nav row = number tile + heading + one-line teaser.** Each row: a **number tile/badge** (the "logo" slot, à la LinkedIn), the **doc heading** as title, and **one quiet line of teaser/summary** beneath (e.g. "Why Next.js — and why not Astro"). Two-ish lines tall. Closest to the reference; helps Dana choose what to read without overwhelming.

3. **Sectioned nav (grouped).** The nav is grouped into sections, not a flat list:
   - **OVERVIEW** — the landing/"Start here" page (single entry, top).
   - **DECISIONS** — the numbered ADRs (number tile).
   - **PRAGMATISM & PROCESS** — the non-numbered pieces (headline pragmatism calls, the "building with AI/BMAD" way-of-working doc). Different glyph (◆) instead of a number.
     The grouping itself tells a story about how Zac thinks (decisions vs. judgement calls vs. process).

**Why:** all three serve "calmer, content-first, reads as judgement not output" while keeping the nav considered. The sectioning resolves the ADR-vs-non-ADR mismatch the PRD's doc set creates (framework decision + pragmatism calls + AI/BMAD piece aren't all numbered ADRs).

**Open follow-ups:** mobile collapse pattern for the left nav; Entry-link placement on front-of-house (FR-10); Console easter-egg art/copy (FR-11).

### 2026-06-24 — Entry link placement: bottom-left page chrome (FR-10)

The Entry link lives in the **bottom-left corner of the page-level chrome** — out on the gradient background, **outside the dark content shell / hero card entirely** (ref arrow in `imports/entry-link-placement-reference.png`). It is part of **`layout.tsx`**, so it appears on **every page**. It mirrors the theme-toggle moon icon that sits in the **top-left** of the same gradient margin — the two bookend the page furniture, top-left toggle / bottom-left "how this is built" link. Maximally understated: in the periphery, on the chrome, never touching the flash or the content. Realises FR-10's "discoverable but not attention-grabbing."

**Mobile reconciliation (decision):** the page chrome is near-zero on mobile (content card has only `p-2` padding — almost no gradient margin), so the bottom-left chrome link has nowhere to sit. On mobile the Entry link is **relocated into the mobile menu drawer** (the existing vaul `MobileMenu`) as a quiet item set apart from the primary nav — consistent with where all nav lives on mobile, and still understated. [ASSUMPTION — confirm with Zac.]

**Design constraints to honour (for DESIGN.md):**

- The link sits on the page **gradient** (dark: cyan→gold top-to-bottom; light: differs). Bottom-left in dark mode overlaps the **gold/yellow** region → link text needs deliberate contrast handling in BOTH themes; cannot assume a flat background. Likely a subdued but legible treatment (e.g. text-primary at reduced weight/opacity with sufficient contrast), verified against both gradients.
- Keyboard-accessible, real `<a>`/`<Link>`, works in static export (FR-10 consequence).

### 2026-06-24 — Backroom layout: near-full-bleed, ~28px frame (mock approved)

After reviewing `backroom-mock.html`, Zac approved the Backroom direction with **one adjustment**: the Backroom shell should be **expansive — fill the viewport behind a ~48px margin/frame** all round, NOT a centred max-width card. (Started at 28px; raised to **48px** so the page chrome — the top-left theme toggle and the bottom-left Entry link — has breathing room and doesn't crowd the shell edge.) This is a deliberate spatial contrast with **front-of-house** (which stays a contained, centred max-width card): the Backroom _opens up_ — reinforcing "you've stepped into a different, calmer reading room with room to read." The thin gradient frame is all that remains of the page chrome.

- Two-pane grid inside the frame: **~320px nav | flexible content**. Content reading column is capped (~64ch, left-aligned) for readability even though the pane is wide.
- No conflict with the Entry link: that's a **front-of-house-only** element (the door _in_), so it keeps front-of-house's generous gradient margin. Inside the Backroom there is no Entry link — only `◀ back to the site`.

**Approved as-is (Zac "quite like what you've put together"):** the LinkedIn-style nav row (cyan number tile + heading + teaser, selected = cyan-tint bg + left accent bar), the gold ★ Overview tile and gold ◆ pragmatism glyph, the sectioned nav, the content-pane treatment (eyebrow + Permanent-Marker title + gold pragmatism callout + syntax-highlighted code).

**Mobile calls — CONFIRMED by Zac (2026-06-24):** the Backroom mirrors the main site's mobile pattern exactly.

- On mobile the Backroom's **left nav panel is hidden inside the portal/shell** — the same way the main site hides its `lg:`-only sidebar nav below the `lg` breakpoint.
- The **entire sectioned nav** (`◀ back to the site`, OVERVIEW, DECISIONS, PRAGMATISM & PROCESS, with the number-tile / glyph rows) **renders into the existing vaul hamburger drawer** (`MobileMenu`) — the same drawer mechanism front-of-house already uses (ADR 0018). Content goes full-width behind it.
- The **Entry link** (front-of-house) likewise becomes a quiet item inside that drawer on mobile (where there's no page chrome for it).
- The desktop two-pane shell only applies at the site's `lg+` breakpoint; below it, single-column content + drawer.
- Not shown in `backroom-mock.html` (desktop mock only) — explicitly a spec note for architecture, per Zac.

Mock promoted candidate: `.working/backroom-mock.html` (+ `backroom-mock.png`).

### 2026-06-24 — Console easter egg: ASCII wizard + casual invite (FR-11)

Direction chosen by Zac (rejected the 4-lane menu in favour of his own): **an ASCII-art wizard** with **casual, informal, self-deprecating copy** that invites the curious dev in. Seed line from Zac:

> "Ah! I see you're also a tech wizard... well, while you're here you might as well come in."

- **Tone:** warm, playful, conversational — dodges the cringe/flex risk by being self-deprecating and inviting rather than boastful. Fits PRD §10 "charming, not cringe."
- **Art:** a compact ASCII wizard (pointy hat + face + beard + robe), console-friendly width.
- **Links (FR-11 requires both):** → the Backroom route, → the GitHub repo ("the source"). Repo is public (PRD OQ#6 resolved).
- **Optional closing wink:** a one-liner that the whole site (docs included) was built with Claude Code + BMAD — ties the egg back to the "AI-augmented way of working" job (but keep it light; v1 is a single static print + links, no multi-stage sequence per PRD).
- **Mechanism (from PRD FR-11):** emit on page load via `console.*`; browsers retain pre-open messages and show them when devtools opens later. Dev-tools-detection re-emit only as evidence-driven fallback. (UX owns art/copy; mechanism is architecture/story.)

Draft artifact: `.working/console-egg-mock.html` (console-styled preview).
