DECISIONS

Framework: Next.js

The first real fork in the road — and the one most worth explaining.

The site was a Gatsby app. The honest question wasn't "what's newest?" but "what gets this shipped, stays maintainable for one person, and exports to pure static HTML?" Three candidates made the shortlist: stay on Gatsby, move to Next.js, or jump to Astro.

Why not Astro

Astro is the better content framework on paper — and that's exactly why it was tempting. But the site isn't only content; it has interactive flourishes (the rotating job title, route-transition animations, a themed drawer) that lean on React's ecosystem. Astro would have meant porting those to islands and re-learning a build I'd use once a year.

The pragmatism call: the right tool isn't the most elegant one — it's the one a sole maintainer can still reason about in eighteen months. Next.js kept the React mental model I already had.

The deciding constraint

Static export was non-negotiable (the site is hosted as a static bundle). Next's output: 'export' closes the door on server features by design, which is a feature here, not a limit — it makes the whole site provably build-time.

// next.config.ts const config: NextConfig = { output: 'export', // no server runtime — everything prerenders to out/ images: { loader: 'custom', loaderFile: './src/image-loader.ts' }, };

Read alongside the border-colour guard (09) and deferring the polish, this is the throughline: decisions that compound rather than constrain.
