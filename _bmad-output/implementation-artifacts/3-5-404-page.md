# Story 3.5: 404 page

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a site visitor who hits an unknown URL,
I want the existing not-found page,
so that a wrong link behaves exactly as it does today.

## Context & purpose (read first)

This is the **fifth and final story of Epic 3** (Content Pages at Parity) — and the last page
to port before Epic 4's cutover. The themed, persistent, fully-navigable shell exists (Epics
1–2 done) and **3.1 (Home), 3.2 (About Me), 3.3 (Resume), and 3.4 (Content) are done** — the
Epic-3 house pattern is fully established: a **Server-Component page** that composes content +
sets per-page `metadata`, with **`'use client'` only on interactive leaves**, all rendered as
`children` inside `<ContentTransition>` in the root `layout.tsx`. This story does the **404 / not-found**
catch-all.

This is the **simplest page in the entire migration** — a **zero-interactivity, no-image,
no-dependency, two-element** page (one `<h1>`, one `<p>`). Every component is a Server Component.
There is **no new component, no image, no `globals.css` change, no new dependency**. The entire
deliverable is **one new file** — `src/app/not-found.tsx` — plus tracking.

**The one genuine engineering decision on this page is structural, not visual: which Next.js
convention to use, and how the title survives static export.** Get the convention right and the
markup is a 30-second verbatim port. Get it wrong and you either lose the persistent shell or ship
Next's default grey 404. **Read Dev Note "The not-found convention decision (settled)" and Dev Note
"Static-export 404 — the two gotchas, and why they don't bite us" before writing a line.**

The single source of visual/behavioural truth is the **live site** (`zackerthehacker.com`) and the
archived Gatsby file (`archive/src/pages/404.js`). This is a **byte-for-byte parity port** (NFR1
zero-visual-regression, NFR2 zero-functional-regression, NFR7 preserve quirks). Content stays
**frozen** — the casual `...the sadness...` copy and the `404: Not found` / `404: Not Found`
title-vs-heading casing quirk (see below) remain **Project Ariadne's** domain, **not Theseus's** (do
**not** "update", "correct", or "tidy" them).

After this: **Epic 3 is complete** — only Epic 4 (parity sign-off + cutover) remains.

**3.5's job — port the not-found page at parity:**

- **Create `src/app/not-found.tsx`** — the **standard App Router root not-found convention** (a
  Server Component), porting `archive/src/pages/404.js` verbatim. This renders inside the existing
  root `layout.tsx` shell (so it inherits the sidebar/nav/content-pane chrome and the
  `<ContentTransition>` route animation) — exactly as the Gatsby 404 rendered inside the wrapped
  Layout today.
- **Do NOT use the `global-not-found.js` convention** (experimental, would bypass the root layout
  and lose the shell, and is actively broken on our Next 16.2.x) — see Dev Note.
- **Do NOT use the shared `Heading` atom** — the archive 404 uses a **bespoke inline `<h1>`** with
  larger sizing (`text-4xl sm:text-6xl`), identical to the **home page's** inline `<h1>`. The
  `Heading` atom is `text-3xl xl:text-4xl` (different) — using it would be a visual regression.
- **Apply the ADR 0021 metadata convention via the root-page exception.** `not-found.tsx` lives in
  the **root segment** (same level as `app/page.tsx` / `app/layout.tsx`), so the `%s - Zac Braddy`
  title template does **not** apply to it — it follows the **home-page `title: { absolute: ... }`
  form**, NOT the child-segment `title: '<Page>'` form. (ADR 0021's prose lists "3.5 404" among the
  child-segment pages — that line is **imprecise**: not-found is root-segment, see Dev Note.)
- **Verify the static export emits the CUSTOM `out/404.html`** (with the real heading + templated
  title), not Next's default 404 — the single must-verify gate on this story.

## Dev Notes

### The not-found convention decision (settled — use `not-found.tsx`, NOT `global-not-found.js`)

Next.js 16 offers **two** conventions for the catch-all 404. They are **not** interchangeable here:

| Convention                                                 | Renders inside root `layout.tsx`?                                                  | Status                   | Verdict for 3.5 |
| ---------------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------ | --------------- |
| **`app/not-found.tsx`** (standard)                         | **YES** — wrapped by the root layout + `<ContentTransition>`                       | **stable** (since v13.3) | **✅ USE THIS** |
| `app/global-not-found.tsx` (`experimental.globalNotFound`) | **NO** — bypasses the layout; must return a full `<html>`/`<body>` document itself | **experimental**         | ❌ avoid        |

**Why `not-found.tsx` is correct (and `global-not-found` is wrong) for this site:**

1. **Parity requires the shell.** Today's Gatsby 404 renders **inside** the wrapped `Layout`
   (via `archive/gatsby-browser.js` / `gatsby-ssr.js` `wrapPageElement`) — so it shows the sidebar,
   nav, and content pane, with the `404: Not Found` block centred in the content area. `not-found.tsx`
   renders as `children` of the root `layout.tsx` (inside `<ContentTransition>`), reproducing exactly
   that. **`global-not-found.js` explicitly bypasses the layout** ("Next.js skips rendering and
   directly returns this global page" — it must ship its own `<html>`/`<body>` and re-import global
   styles/fonts), which would render a **bare, shell-less 404** = a hard parity regression.
2. **`global-not-found` is experimental AND actively broken on our exact version.** It requires
   `experimental: { globalNotFound: true }`, and there is an open Next.js issue (#92256) —
   **"global-not-found breaks with any import (infinite dev loop, high CPU) in Next 16.2.x"** — and
   this project is on **16.2.9**. Adopting an experimental, version-broken convention violates the
   [[theseus-idiomatic-next-principle]] (build Next-native on **stable** primitives) and NFR6.
3. **No multiple root layouts / no dynamic root segment.** `global-not-found` exists for apps that
   _can't_ compose a 404 from `layout.js` + `not-found.js` (multiple root layouts, or a
   `app/[country]/layout.tsx` dynamic root). This site has a **single static root layout** and flat
   static routes — the exact case `not-found.js` is designed for.

**Do not create `app/404/page.tsx` either** — that would register a literal `/404` _route_, not the
unmatched-URL catch-all. The convention is the reserved filename `app/not-found.tsx`.

### Static-export 404 — the two gotchas, and why they don't bite us (read before AC #4)

There are two historical Next.js static-export (`output: 'export'`) issues around custom 404s. Both
are about **dynamic routes** — which **this site does not have** — so neither should bite, but the
AC #4 build check exists to **prove** it empirically:

1. **"Root `not-found.tsx` not compiled to `404.html` under static export" (issue #48227).** This was
   a Next 13 bug, **fixed** (PR #52526). In current Next (15/16) a root `app/not-found.tsx` **does**
   generate a custom `out/404.html` under `output: 'export'`. Netlify automatically serves `404.html`
   for unmatched URLs, so the production behaviour matches the AC.
2. **"`not-found.tsx` doesn't build with `output: export` in _dynamic_ routes" (issues #54239 / the
   i18n `[lang]` cases).** These failures are specific to **dynamic-segment** layouts (`app/[lang]/…`)
   where the not-found can't be statically resolved. This site has **no dynamic segments** — every
   route is a flat static path — so this class of failure does not apply.
3. **Metadata-stripped-on-hydration (issue #61236)** is likewise specific to **404s returned from
   dynamic paths** (the title is correct in the server HTML but removed on client hydration). For a
   fully-static site with no dynamic paths, the static `out/404.html` retains the templated `<title>`.

**Net:** root `not-found.tsx` → custom `out/404.html` with the templated title is the expected,
supported behaviour on Next 16.2.9 for this static, dynamic-route-free site. **AC #4 makes you open
`out/404.html` and confirm it** (custom heading + templated title + shell markup, not Next's default).
If — contrary to expectation — it emits Next's default 404, that's a genuine non-obvious
reconciliation → investigate and capture as ADR 0025 (AC #5) before claiming done. Do **not** silently
ship the default 404.

### Why the root-page exception, not the child form (the title gotcha)

ADR 0021 established two title shapes for Epic 3:

- **Child-segment pages** (`/about-me`, `/resume`, `/content`): `title: '<Page>'` + the root
  `%s - Zac Braddy` template → single-suffix `<title>` (e.g. `Resume - Zac Braddy`). Used by 3.2–3.4.
- **Root-segment page** (the home page, `app/page.tsx`): the template does **NOT** apply to the
  segment where it's defined (Next 16 behaviour), so the home page sets `title: { absolute: 'Home - Zac Braddy' }`.

**`not-found.tsx` is in the ROOT segment** — same level as `app/layout.tsx` and `app/page.tsx`, **not**
a child route folder. So it follows the **home-page `{ absolute }` form**, **not** the child-segment
form. Using `title: '404: Not found'` (child form) would render a **bare `<title>404: Not found</title>`
missing the ` - Zac Braddy` suffix** — failing AC #2.

> **Note on ADR 0021's prose:** its consequences list groups "(3.2 About Me, 3.3 Resume, 3.4 Content,
> 3.5 404)" as pages that "live at their own child segments … where the template does apply." That
> phrasing is **imprecise for 3.5** — not-found is root-segment, so it takes the `{ absolute }`
> exception (the same one the prose already carves out for the home page). This story applies the
> exception correctly; no ADR edit is needed (the convention — single-suffix title — is unchanged;
> only the _mechanism_ differs, exactly as it already does for the home page).

**Verify the rendered `<title>` in `out/404.html`** (AC #4) — it is the authoritative check that the
`{ absolute }` form produced `404: Not found - Zac Braddy`.

### Preserve the title/heading casing quirk (`404: Not found` vs `404: Not Found`)

The archive has a deliberate-looking inconsistency, preserved **verbatim** (NFR7):

- **Document title** (`Seo title="404: Not found"`): **lowercase "found"** → templated to
  `404: Not found - Zac Braddy`.
- **Visible `<h1>`** (`404: Not Found`): **capital "Found"**.

**Keep both exactly as-is** — title lowercase, heading capital. Do **NOT** "fix" the casing to make
them match — that is an **editorial/content** change (Project Ariadne's domain, like the 3.4 YouTube
casing carve-outs), and content carve-outs require Zac's **explicit** approval (none given for 3.5).
This differs from ADR 0021's doubled-`<title>` fix, which corrected an **obvious mechanical bug** (a
react-helmet template artefact, not authored copy); the `found`/`Found` casing is **authored copy**,
so it stays frozen. Logged as a quirk for Ariadne / the Story 4.1 gate (Task 3). _(If Zac wants the
casing aligned, that's a one-line carve-out — but it is his call to make, not the dev agent's.)_

### Do NOT use the `Heading` atom (the heading-styling gotcha)

The archive 404 heading is a **bespoke inline `<h1>`**, **not** the shared `Heading` atom:

```jsx
// archive/src/pages/404.js — the 404 heading (same styling as the home page <h1>):
<h1 className="font-fancy-heading text-4xl text-secondary sm:text-6xl mb-4">
  404: Not Found
</h1>
```

Compare the shared `Heading` atom (`src/components/atoms/heading.tsx`), used by Resume/Content:

```tsx
<h1 className={`${className} font-fancy-heading text-3xl xl:text-4xl`}>
  {children}
</h1>
```

These are **different sizes** (`text-4xl sm:text-6xl` vs `text-3xl xl:text-4xl`) and the 404 also
uses `text-secondary` + `mb-4` inline. The home page (`src/app/page.tsx`) already renders this exact
inline `<h1>` — the 404 mirrors it. **Reusing `Heading` here would be a visual regression** — keep
the inline `<h1>` verbatim. (This is the one place on Epic 3 where reusing a "matching" atom is
**wrong**; the markup only looks similar.)

### OG/Twitter on the 404 (minimal-surface decision)

AC #2 requires **only** the `<title>`. The page is a 404 — **Next automatically injects
`<meta name="robots" content="noindex">` on 404-status pages**, so it is never indexed or shared.
For parity context: the archive's `Seo` ran for the 404 too and emitted `og:title`/`twitter:title` =
the **raw, unsuffixed** `404: Not found` (react-helmet's `titleTemplate` applies only to `<title>`,
not to meta tags). So the live 404 has `<title>404: Not found - Zac Braddy</title>` but
`og:title="404: Not found"` (unsuffixed).

**Decision (NFR6 anti-gold-plating):** set **only** `title: { absolute: '404: Not found - Zac Braddy' }`
and let `description`/`openGraph`/`twitter`/OG-image/favicon **inherit** the root-layout defaults — do
**not** re-declare them. Rationale: the AC requires only the title; the page is `noindex` and never
shared, so a divergent og:title is immaterial; re-declaring meta for an unshared 404 adds surface for
zero parity-meaningful gain. The inherited og:title (`Zac Braddy - CV website`) differs from the
archive's `404: Not found`, but on a noindex page this is **not a perceptible/functional regression** —
log it as an immaterial, expected delta for the Story 4.1 gate. _(Open question flagged to Zac below —
if he wants strict og parity, add `openGraph.title`/`twitter.title: '404: Not found'`; his call.)_

### Why every component is a Server Component (AR14, NFR5)

The 404 has **zero interactivity** — no hooks, no `onClick`, no client APIs, no link. So `not-found.tsx`
is a plain Server Component (the not-found convention is server-by-default, AR14). Do **not** add
`'use client'` "to be safe". (Even the Next docs' `<Link>`-back-home example is interactivity the
archive doesn't have — don't add it.)

### Frozen content — port verbatim (NFR7, [[theseus-content-frozen-ariadne-owns-refresh]])

Epic 3 is like-for-like translation. On this page that means **leaving alone**: the casual
`...the sadness...` copy, the `doesn't exist` phrasing, and the `404: Not found`/`404: Not Found`
casing. These remain **Ariadne's** content refresh, **not Theseus's**. The only intentional changes
are mechanical: JS→TS typing, the `Seo`→`metadata` swap, and `&#39;`→`&apos;` apostrophe escaping

## Decision trail

1. **Use the stable `app/not-found.tsx` convention, NOT `global-not-found.js` (settled).** `not-found.tsx`
   renders inside the root `layout.tsx` shell (parity), is **stable**, and fits a single-static-root-layout
   app. `global-not-found` bypasses the layout (loses the shell → regression), is **experimental**, and
   is **broken on Next 16.2.x** (issue #92256). [[theseus-idiomatic-next-principle]] / NFR6. See Dev Note
   "The not-found convention decision".
2. **Root-page `{ absolute }` title form, NOT the child-segment form (settled — ADR 0021).** not-found
   is in the **root segment**, so the `%s - Zac Braddy` template doesn't apply — use
   `title: { absolute: '404: Not found - Zac Braddy' }` (the home-page mechanism). The child-segment
   `title: '<Page>'` form (3.2–3.4) would drop the suffix. ADR 0021's "3.5 404" prose grouping is
   imprecise; this story applies the exception correctly (no ADR edit needed). See Dev Note.
3. **Preserve the `404: Not found`/`404: Not Found` casing quirk verbatim (settled — NFR7).** Authored
   copy, not a mechanical bug — frozen content, Ariadne's domain; no carve-out approved for 3.5. Logged
   for the 4.1 gate. (Differs from ADR 0021's doubled-`<title>` fix, which corrected a helmet artefact.)
4. **Bespoke inline `<h1>`, NOT the `Heading` atom (settled — parity).** The archive 404 heading uses the
   home-page sizing (`text-4xl sm:text-6xl`), which differs from the `Heading` atom; reusing the atom is a
   visual regression. See Dev Note.
5. **Set only the title; inherit OG/Twitter/description/favicon (settled — NFR6).** The AC requires only
   `<title>`; the page is `noindex`/unshared, so re-declaring meta adds surface for no parity gain. Logged
   the inherited-vs-archive og:title as an immaterial delta. _(Open question to Zac below.)_
6. **Every component is a Server Component (settled — AR14/NFR5).** Zero interactivity → no `'use client'`,
   no link.
7. **No new ADR expected (settled — NFR6).** Faithful port on a stable convention executing ADR 0021. ADR
   0025 **only if** static export fails to emit a custom `out/404.html` and a workaround is forced (0024 is
   the highest existing; 0025 is next).
